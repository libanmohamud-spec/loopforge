Scheduled loop

The overnight docs sweep

Contributed by Matthew Berman

Run this prompt

Each night, review the codebase in full and make sure all documentation reflects the latest changes from the previous day. Update the documentation as needed, then open a pull request with those changes.

Trigger
Nightly schedule
Inspect
Full codebase and the previous day's changes
Verify
Documentation matches the current implementation
Output
A reviewable pull request

Why it works It turns documentation drift into a small, reviewable daily maintenance task instead of a cleanup project.

Goal loop

The architecture satisfaction loop

Contributed by Peter Steinberger

Run this prompt

Refactor until you are happy with the architecture. After each significant step, live-test the system, run autoreview, and commit. Track progress in /tmp/refactor-{projectname}.md.

Goal
A coherent architecture, not a fixed task count
Checkpoint
Live test, autoreview, and commit each significant step
Memory
/tmp/refactor-{projectname}.md
Stop
The architecture is satisfactory and checks pass

Why it works It keeps a broad refactor inspectable by pairing every major move with live evidence, review, a commit, and external memory.

Loop anatomy

A loop needs more than repetition.

The best agent workflows make progress visible. They tell the agent when to run, what evidence counts, where state lives, and when the job is actually finished.

  1. 01

    Trigger

    A schedule, event, failure, or goal starts the loop.

  2. 02

    Move

    A bounded action advances the work without hiding the diff.

  3. 03

    Proof

    Tests, review, or live behavior show whether the move worked.

  4. 04

    Memory

    A file, commit, issue, or record carries context forward.

  5. 05

    Stop

    A concrete condition prevents motion from becoming churn.
